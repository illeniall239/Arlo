1. WHAT IS WEB SCRAPING?
================================================================================

Web scraping is the automated extraction of data from websites. Humans browse
websites visually — a scraper does the same thing programmatically: it fetches
HTML, parses the structure, and extracts the specific data it needs.

Common use cases:
- Price monitoring (e-commerce competitors)
- Job listing aggregation
- Research data collection
- Real estate data (Zillow, Rightmove)
- Financial data (stock prices, earnings)

LEGAL NOTE: Scraping publicly accessible data is generally legal in most
jurisdictions (see hiQ v. LinkedIn, 2022). Always respect robots.txt and
rate limits.

================================================================================
2. HOW THE WEB WORKS (HTTP FUNDAMENTALS)
================================================================================

Every web page interaction is an HTTP request/response cycle:

  [Browser/Scraper]  --HTTP GET--> [Web Server]
  [Web Server]       --HTML/JSON-> [Browser/Scraper]

HTTP Headers (sent with every request):
  User-Agent     — Identifies the client (browser, bot, scraper)
  Accept         — What content types the client accepts
  Cookie         — Session identifiers
  Authorization  — API keys, Bearer tokens

  → Anti-bot systems inspect these headers. A scraper with User-Agent:
    "python-requests/2.31.0" is instantly flagged.

HTTP Status Codes:
  200 OK         — Success
  403 Forbidden  — Likely bot detection
  429 Too Many Requests — Rate-limited
  503 Service Unavailable — Blocking or overloaded

TLS / HTTPS Fingerprinting:
  Each HTTP client has a unique "TLS fingerprint" based on the cipher suites
  it negotiates. Python's ssl module fingerprint differs from Chrome's.
  Anti-bot services (Cloudflare, DataDome) check this at the connection level —
  before even seeing your User-Agent.
  curl_cffi impersonates Chrome's exact TLS fingerprint. This is our Tier 1
  defense: the very first handshake looks like Chrome.

3. TYPES OF WEB PAGES AND WHY IT MATTERS
================================================================================

TYPE 1: STATIC HTML
  - Server returns complete HTML directly
  - Data visible in curl output
  - Best path: curl_cffi → parse HTML → embedded extractors or Jina+Gemini

TYPE 2: SERVER-SIDE RENDERED WITH EMBEDDED DATA
  - Server renders full HTML, but also pre-renders the full dataset as JSON
    inside <script> tags for client-side hydration
  - Examples: Indeed (window.mosaic), Next.js apps (__NEXT_DATA__), Glassdoor (apolloState)
  - Best path: curl_cffi → embedded extractor → done (no AI needed)
  - This is the golden path: fastest, cheapest, most accurate

TYPE 3: CLIENT-SIDE RENDERED (SPA) / INFINITE SCROLL / CLOUDFLARE-PROTECTED
  - Shell HTML only, content rendered by JavaScript; or infinite-scroll content;
    or protected by Cloudflare JS challenges
  - curl returns placeholder HTML or a challenge page
  - Best path: Jina Reader — handles JS rendering, Cloudflare, and infinite scroll
    internally on its own servers. Returns clean markdown with all visible content.
  - No Playwright dependency on our side (cloud-deployable, ~200MB Docker image)

4. WEB SCRAPING TECHNIQUES
================================================================================

4.1 HTML PARSING
  Web pages are HTML trees. Scraping means navigating the tree to extract values.
  CSS Selectors: page.css(".job-title").text
  XPath: //div[@class="job-card"]/h2/text()
  In Arlo we do NOT generate CSS selectors — Gemini reads plain text instead.

4.2 EMBEDDED DATA EXTRACTION (our primary path for modern sites)
  Many modern sites written in React/Next.js/Nuxt pre-render their entire
  dataset as JSON inside <script> tags before sending HTML to the browser.
  This is for fast page hydration (client doesn't need to re-fetch).

  For us, this is a goldmine: the structured data is already there.
  No need for AI, no need for selectors, no need for browser rendering.

  Common patterns:
    <script id="__NEXT_DATA__" type="application/json">{...}</script>
    <script>window.__NUXT__ = {...}</script>
    <script>window.mosaic.providerData["mosaic-provider-jobcards"] = {...}</script>
    <script type="application/ld+json">[{"@type": "JobPosting", ...}]</script>

4.3 JINA READER (our primary fallback)
  Jina Reader (r.jina.ai/{url}) is a service that fetches any URL and returns
  clean markdown — stripping navigation, ads, boilerplate, and rendering JS.
  It handles Cloudflare and other bot protection internally.

  Why Jina over raw HTML → Gemini:
  - Clean markdown is far smaller than raw HTML (10x compression typical)
  - Removes all the noise Gemini would otherwise have to ignore
  - Jina handles JS rendering transparently — we don't need to know if a site is SPA
  - Markdown links are preserved: [Job Title](https://...)

4.4 PAGINATION
  a) URL Parameter:
     Most sites: ?page=1 → ?page=2. We scan HTML/markdown for rel="next"
     links, "Next" buttons, and URL patterns.

  b) Infinite Scroll:
     No page URLs — content loads as you scroll. We use Playwright and
     repeatedly scroll to document.body.scrollHeight until height stops
     growing or target record count is reached.

  c) Cursor/API-based:
     Some sites expose JSON APIs with cursor pagination. Tier 0 handlers
     use these directly when we know the API structure.

4.5 RATE LIMITING (ETHICAL SCRAPING)
  Best practices:
  - 1-2 second delay between requests to the same domain
  - Respect robots.txt
  - Use Jina/Tier 0 APIs where possible (they cache aggressively)

5. ANTI-BOT SYSTEMS AND HOW TO BYPASS THEM
================================================================================

5.1 USER-AGENT DETECTION
  Detection: Checks User-Agent for known bot strings.
  Counter: curl_cffi sends a realistic Chrome User-Agent automatically.

5.2 TLS FINGERPRINTING (JA3/JA4)
  Detection: TLS handshake reveals cipher suites. Python's ssl module
  has a different fingerprint from Chrome.
  Counter: curl_cffi impersonates Chrome's exact TLS fingerprint.
  This is our first line of defense — handled before any HTML is seen.

5.3 BROWSER FINGERPRINTING
  Detection: JavaScript checks navigator.webdriver, canvas fingerprint,
  WebGL renderer, screen resolution, hardware concurrency, timezone, etc.
  Counter: Playwright with stealth patches for browser-render path.
  Best counter: Jina Reader — we never run the browser ourselves.

5.4 BEHAVIORAL ANALYSIS
  Detection: Bots click instantly, scroll perfectly, have no mouse movement.
  Counter: fetch_with_scroll adds realistic delays between scrolls (1500ms).

5.5 CLOUDFLARE
  Checks: JS challenges, Turnstile, IP reputation, bot ML models.
  Counter: Jina Reader handles Cloudflare transparently (Tier 3).
  For sites where curl_cffi fails and JS rendering is needed, Jina is
  always tried as the final fallback before giving up.

5.6 IP-BASED BLOCKING
  Counter: Jina's infrastructure handles IP diversity for us.
  Direct scraping: proxies can be added via settings, not yet implemented.

6. THE FETCHING STACK — curl_cffi, Jina Reader
================================================================================

TIER 1 FETCH: curl_cffi (fetch_raw)
  Library: curl_cffi with impersonate="chrome120"
  What it does:
    - Opens an HTTP/2 connection with Chrome's exact TLS fingerprint
    - Sends realistic browser headers (Accept, Accept-Language, etc.)
    - Follows redirects
    - Returns raw HTML
  When it succeeds: Static HTML sites, sites without aggressive bot checks
  When it fails: JS-heavy SPAs (returns placeholder HTML), Cloudflare-protected sites
  Speed: Very fast (< 1 second for most sites)

TIER 2 FETCH: Jina Reader (fetch_jina)
  API: GET https://r.jina.ai/{url}
       Authorization: Bearer {JINA_API_KEY}  (optional, raises rate limits)
  What it returns: Clean markdown of the page content
  What Jina does internally:
    - Renders JavaScript (Headless Chrome on Jina's servers)
    - Removes navigation, ads, sidebars, footers
    - Converts links to [text](url) format
    - Handles Cloudflare, TLS fingerprinting, and bot protection
    - Returns clean, structured text
  When it's used: Any time embedded extraction fails (fast path or fallback)
  Speed: 3-8 seconds (Jina renders server-side, has its own caching)

  WHY JINA REPLACED PLAYWRIGHT:
    Playwright (Chromium) added ~1GB to the Docker image. Jina handles JS rendering,
    Cloudflare, and infinite-scroll transparently on its own infrastructure. We get
    the same coverage with a ~200MB image and no browser management overhead.
    Jina is the correct universal fallback — not a workaround.

fetch_plain(url):
  Plain httpx GET with minimal headers. Used for JSON APIs (site_apis.py) and
  RSS/Atom feeds. No fingerprint tricks needed — these are APIs expecting
  programmatic access.

7. THE EXTRACTION PIPELINE — THREE TIERS
================================================================================

The job_runner._run_paginated_extraction() function implements this flow:

```
for page_num in range(1, max_pages + 1):

    [TIER 0] Try site-specific API (site_apis.py _SITE_MATCHERS)
             If match → fetch JSON directly → records → done

    [TIER 1] curl_cffi raw HTML fetch
             → extract_embedded(html) → if records → use them → paginate URL loop
             → if no records → fall through to TIER 2

    [TIER 2] fetch_jina(url) → strip to plain text → Gemini extract
             → (records, has_next_page, next_button_selector)
             → _validate_urls: remove any URL Gemini hallucinated
             → if has_next_page → continue URL loop
             → else → break
```

DESIGN PRINCIPLE: Each tier is tried only if the previous tier failed.
The embedded extractor is always attempted before any AI call because:
  - It is faster (no network round-trip to Gemini)
  - It is cheaper (no API tokens consumed)
  - It is more accurate (exact data, not AI interpretation of text)

8. EMBEDDED DATA EXTRACTION — PRE-RENDERED SCRIPT TAGS
================================================================================

File: backend/app/services/embedded.py

Many modern web apps (especially React/Next.js/Nuxt.js) pre-render their data
into the HTML as JSON to enable fast client-side hydration. We extract this
directly — it is the most reliable extraction method available.

EXTRACTOR 1: window.mosaic (Indeed)
  Indeed's React app stores job card data in:
    window.mosaic.providerData["mosaic-provider-jobcards"]
  Contains: job title, company, location, salary, URL, description snippet
  Extraction: regex to find the JS assignment, json.loads the value

EXTRACTOR 2: apolloState (Glassdoor)
  Glassdoor's GraphQL client caches results in:
    window.__APOLLO_STATE__ = {...}
  Format: flat dict of "Entity:id" keys with __ref cross-references
  Extraction: resolve __ref chains to assemble complete job objects

EXTRACTOR 3: application/ld+json (Schema.org)
  Standard metadata format. JobPosting, Product, Event, Article types.
    <script type="application/ld+json">[{"@type":"JobPosting", ...}]</script>
  Flat fields: title, hiringOrganization, baseSalary, jobLocation, description
  Note: Organization and Product blocks are filtered out — they are site metadata
  _looks_like_content() gate: rejects records if <50% have content-indicating
  fields (description, salary, price, location, date, etc.)

EXTRACTOR 4: __NEXT_DATA__ (Next.js)
  Every Next.js app injects its server-side props into:
    <script id="__NEXT_DATA__" type="application/json">{...}</script>
  Contains: pageProps → usually the full API response for the current page
  Extraction: recursively find the largest list of dicts (_best_list heuristic)
  Then apply flatten_record() — Next.js data often has deeply nested objects

EXTRACTOR 5: __NUXT__ / __NUXT_DATA__ (Nuxt.js)
  Nuxt 2 uses: window.__NUXT__ = {...}
  Nuxt 3 uses: <script id="__NUXT_DATA__" type="application/json">[...]</script>
  Similar hydration pattern to Next.js — extract largest list of dicts

_looks_like_content(records) QUALITY GATE:
  Problem: Organization and Product JSON-LD blocks were being returned as
  "2 records" — preventing the JS-heavy browser-render path from running.
  Solution: Check that ≥50% of records have at least one field from the set:
    {description, salary, price, location, url, href, date, posted,
     company, employer, title, name, email, phone, address}
  If <50% qualify, the batch is rejected and the next tier is tried.

9. THE AI EXTRACTION STAGE — GEMINI ON PLAIN TEXT
================================================================================

File: backend/app/services/ai_pipeline.py

WHY TEXT INSTEAD OF CSS SELECTORS:
  The original design had Gemini generate CSS selectors in a "planning" stage,
  then Scrapling execute them. This approach had fundamental problems:
  - CSS selectors are brittle — one site redesign breaks them
  - Gemini's selector generation quality was inconsistent
  - The planning-then-execute loop doubled API latency
  - Modern JS-heavy sites may render differently than their HTML suggests

  New approach: Gemini reads the visible text content of the page and extracts
  records directly. Like a human reading a job listing page and writing down
  the jobs in a spreadsheet. No selectors. No DOM traversal.

INPUT PREPARATION:
  _html_to_text(html):
    - Removes <script>, <style>, <nav>, <header>, <footer> tags entirely
    - Keeps <a href="..."> → appends [LINK:url] after the text
    - Strips all other HTML tags, leaving only text + link markers
    - Compresses whitespace

  _markdown_to_text(md):
    - Converts [text](url) → text [LINK:url]
    - Removes markdown formatting (##, **, __, `code`, etc.)
    - Keeps the readable text + link references

  _strip_tracking_params(text):
    - URLs with >150-char query strings are truncated to path only
    - Prevents token waste on utm_* tracking garbage

GEMINI PROMPT STRUCTURE:
  System: "You are a web data extraction expert. Extract ALL records matching
  the user's goal from the page content below. Return JSON with:
    {
      'records': [{field: value, ...}, ...],
      'has_next_page': true/false,
      'next_button_selector': 'CSS selector or null'
    }
  All field values must be plain scalars (strings, numbers, booleans).
  Never produce nested objects or arrays — join list values with ', '.
  Ignore navigation menus, sidebars, breadcrumbs, headers, footers, and
  category/tag listings unless the goal explicitly asks for them.
  Only extract items from the main content area."

  User: "Goal: {goal}\n\nPage content:\n{plain_text}"

  response_mime_type="application/json" — guaranteed valid JSON from Gemini

RETRY LOGIC:
  Gemini occasionally returns 503 UNAVAILABLE or 500 INTERNAL.
  Exponential backoff: 1s, 2s, 4s — up to 3 retries per extraction call.

PAGINATION SIGNAL:
  Gemini returns has_next_page and next_button_selector.
  job_runner cross-checks with _scan_next_page_url() (regex-based next-link finder)
  and uses either signal to decide whether to continue the URL loop.

11. DATA FLOW (END-TO-END)
================================================================================

1. User fills form: url="https://books.toscrape.com", prompt="Get all book titles, prices and ratings",
   mode=crawl, max_pages=5

2. Frontend: POST /jobs {url, prompt, fetcher_type="auto", max_pages=5}

3. Backend: INSERT ScrapeJob(status=PENDING), enqueue BackgroundTask(run_job, id)
   Returns 201 {id: "uuid", status: "pending"}

4. Frontend: navigates to /jobs/{id}, opens EventSource to GET /jobs/{id}/stream

5. BACKGROUND JOB RUNS:

   Stage 0 — Tier 0 check:
     Does url match any _SITE_MATCHERS? → books.toscrape.com: no match
     SSE: "Starting scrape..."

   Page 1 — Tier 1 (curl_cffi):
     fetch_raw("https://books.toscrape.com") → static HTML
     extract_embedded(html) → 0 records (no script data)
     → fall through to Tier 2
     SSE: "Fetching via Jina Reader..."

   Page 1 — Tier 2 (Jina + Gemini):
     fetch_jina(url) → clean markdown with [text](url) links
     _build_link_map(markdown) → {title_lower → absolute_url} for all 20 books
     _markdown_to_text(markdown) → plain text with [LINK:url] markers (underscores preserved)
     Gemini extract → 20 book records {title, price, rating, url}
     _validate_urls(records, link_map.values()) → all 20 URLs confirmed in link_map
     SSE: "Page 1: 20 records"
     → has_next_page=True → fetch /catalogue/page-2.html

   Pages 2-5 — same Tier 2 path, 20 records each

   Final Normalize + Save:
     flatten_record on each record (no nested fields for this site)
     INSERT ScrapeResult(structured_data=[100 records], row_count=100)
     UPDATE ScrapeJob(status=COMPLETED)
     SSE: {type: "done", result_id: "uuid", rows: 100}

6. Frontend receives "done" event → fetches GET /jobs/{id}/results
   → renders ResultsTable with 100 rows
   → ExportButtons: JSON / CSV download

12. DATABASE DESIGN
================================================================================

TABLE: scrape_jobs
  id              TEXT PRIMARY KEY   — UUID v4 (no enumeration via sequential IDs)
  prompt          TEXT NOT NULL      — User's natural language goal
  url             TEXT NOT NULL      — Target URL (normalized: scheme, whitespace stripped)
  status          TEXT NOT NULL      — pending/running/completed/failed/cancelled
  fetcher_type    TEXT               — auto/Fetcher/StealthyFetcher/DynamicFetcher
  max_pages       INTEGER DEFAULT 10 — Max pages for crawl mode
  batch_id        TEXT               — UUID grouping batch-submitted jobs
  prompt_hash     TEXT               — SHA256(prompt+url) stored but dedup check removed
  created_at      DATETIME
  started_at      DATETIME
  completed_at    DATETIME
  error           TEXT               — Error message if failed

TABLE: scrape_results
  id              TEXT PRIMARY KEY
  job_id          TEXT NOT NULL → FK scrape_jobs.id (CASCADE DELETE)
  structured_data TEXT NOT NULL — JSON array of flattened records
  row_count       INTEGER
  schema_detected TEXT          — JSON array of field names
  created_at      DATETIME

WHY ASYNC SQLALCHEMY:
  FastAPI is async (asyncio). Synchronous DB calls block the event loop —
  no other requests can be served while waiting for DB I/O.
  Solution: SQLAlchemy 2.0 async engine + aiosqlite driver.
  All queries use: async with AsyncSession() as db: await db.execute(...)

WHY UUID PRIMARY KEYS:
  Sequential integer IDs allow enumeration (/jobs/1, /jobs/2...).
  UUIDs are opaque and non-guessable.

13. REAL-TIME COMMUNICATION — SSE
================================================================================

WHY WE NEED REAL-TIME:
  A scrape job takes 5-60 seconds. Users need live progress:
  "Analyzing page...", "Browser rendering...", "124 records found", "Done."

SSE vs WEBSOCKETS:
  Job updates are server → client only. SSE is the right choice:
  - Unidirectional: server pushes, client listens
  - Plain HTTP: no upgrade handshake, no special proxy configuration
  - Native browser EventSource API with auto-reconnect
  - WebSockets add bidirectional complexity that's not needed here

HOW SSE WORKS IN THIS APP:
  1. Client opens EventSource("http://localhost:8000/jobs/{id}/stream")
  2. FastAPI streams text/event-stream:
       data: {"type": "status", "message": "Analyzing page..."}\n\n
       data: {"type": "progress", "rows_found": 42}\n\n
       data: {"type": "done", "result_id": "uuid", "rows": 124}\n\n
  3. Each \n\n terminates an event — browser fires onmessage
  4. Client closes EventSource on done/error event

KEEPALIVE:
  HTTP proxies close idle connections after ~30-60s.
  SSE endpoint sends a comment ping every 30s:
    : keepalive\n\n  (lines starting with : are SSE comments, ignored)

REPLAY BUFFER (with Redis):
  When UPSTASH_REDIS_URL is set, SSEManager uses Redis Pub/Sub.
  sse_buf:{job_id} stores all events (RPUSH, TTL 5 min, max 500).
  A client connecting mid-job gets all prior events replayed first.

14. BACKGROUND JOB ARCHITECTURE
================================================================================

WHY BACKGROUND TASKS:
  A scrape job takes 5-60 seconds. Blocking the HTTP response for that long
  is unacceptable. Accept immediately (201), run in background, stream status.

FASTAPI BACKGROUNDTASKS:
  background_tasks.add_task(run_job, job_id)
  Runs in the same process on the same asyncio event loop after response is sent.

ASYNC + BLOCKING CALLS:
  Playwright, curl_cffi, and Jina are all I/O-bound operations.
  Playwright requires its own event loop (ProactorEventLoop on Windows).
  Solution: run_in_executor (ThreadPoolExecutor) for blocking calls.
  fetch_with_scroll specifically creates a fresh asyncio.run() in its thread.

COOPERATIVE CANCELLATION:
  FastAPI BackgroundTasks have no built-in cancel.
  CancellationRegistry: dict[job_id → asyncio.Event]
  - Between each pipeline stage: check cancel_event.is_set()
  - DELETE /jobs/{id}: sets the event
  - Background task raises and marks job CANCELLED

RETRY LOGIC:
  Gemini API calls: exponential backoff 1s, 2s, 4s (up to 3 retries)
  Network fetches: one retry on timeout/connection error
  No retry on 403 (blocked) or 404 (not found)

  POST /jobs/ — 10/minute per IP
  POST /jobs/batch — 5/minute per IP
  Exports — 30/minute per IP

  201 Created — new job/batch created
  204 No Content — job cancelled
  404 Not Found — job/result not found
  422 Unprocessable — validation failure (Pydantic, auto-generated)
  429 Too Many Requests — rate limit hit

17. UNIVERSAL RECORD NORMALIZER
================================================================================

File: backend/app/utils/normalizer.py

THE PROBLEM:
  Modern sites store rich data in nested JSON. When extracted via embedded
  extractors, records can look like:
    {
      "title": "Senior Engineer",
      "base_salary": {"minValue": 90000, "maxValue": 130000, "unitText": "YEAR"},
      "hiring_org": {"name": "Acme Corp", "@type": "Organization"},
      "skills": ["Python", "Go", "Rust"]
    }

  Storing this in the DB as-is means table cells render as raw JSON.
  The UI shows a 400-char JSON blob in a 240px cell.

THE SOLUTION: flatten_record()
  Applied at two points:
  1. In embedded.py extractors (at source)
  2. In job_runner._execute_pipeline() before DB save (safety net)

FLATTENING RULES (flatten_value):
  Range dict {minValue, maxValue, unitText}:
    → "90,000–130,000/YEAR"

  Range dict {minValue, maxValue} only:
    → "90,000–130,000"

  Named entity dict {name: "Acme Corp", ...}:
    → "Acme Corp"

  Generic dict {key: val, key: val}:
    → "key: val, key: val"

  List ["Python", "Go", "Rust"]:
    → "Python, Go, Rust"

  String with HTML tags "<b>Engineer</b>":
    → "Engineer"

  Recursion depth limit: 3 (prevents infinite recursion in pathological data)

DESIGN PRINCIPLE:
  The normalizer is universal — no special cases per site.
  Any site's nested data is flattened by the same rules.
  New sites "just work" without any normalizer changes.

18. PAGINATION — URL-BASED
================================================================================

THE CORE APPROACH:
  URL-paginated: page 1 at /page-1, page 2 at /page-2 or ?page=2, etc.
    → Loop through URLs up to max_pages, scrape each page independently
    → Standard "crawl" mode

  Infinite-scroll and JS-heavy sites: Jina handles JS rendering internally.
    Jina returns the visible page content as markdown. Pagination still works
    via URL links if the site has them. For true infinite-scroll (no page URLs),
    the Gemini extraction reports has_next_page=False and the loop stops.

HOW PAGINATION WORKS:

  for page_num in range(1, max_pages + 1):
    fetch current_url → extract records → find next URL → update current_url

  _scan_next_page_url(content, base_url):
    Scans HTML/markdown for:
    - <link rel="next" href="...">
    - <a> containing "Next", "→", ">", next
    - URL patterns with page=N, p=N, offset=N incremented by pattern
    Returns the next URL string or None.

  Gemini's has_next_page signal cross-checked with _scan_next_page_url().
  Either signal finding a next page continues the loop.

20. KEY TRADE-OFFS
================================================================================

TRADE-OFF 1: Text extraction vs CSS selector generation
  Old: Gemini plans CSS selectors → Scrapling executes them
  New: Gemini reads plain text directly → extracts records
  Chose: Text extraction
  Gained: No selector maintenance, works on any page structure, no brittle
          DOM-walking, Jina's markdown is noise-free
  Sacrificed: Precise field targeting (text extraction may miss edge cases)
  Verdict: Correct choice — CSS selectors broke on site redesigns constantly

TRADE-OFF 2: Embedded extraction first vs AI first
  Chose: Embedded extraction first (always attempt before AI call)
  Gained: Zero LLM cost for sites with pre-rendered data (majority of
          modern job boards), faster results, higher accuracy
  Sacrificed: Minor added complexity in embedded.py
  Verdict: Essential optimization

TRADE-OFF 3: Jina Reader vs DIY JS rendering (Playwright)
  Chose: Jina Reader as universal Tier 2 fallback
  Gained: Jina handles Cloudflare, ad removal, JS rendering, infinite scroll,
          and link preservation without managing browser infrastructure.
          Docker image ~200MB vs ~1.2GB with Playwright/Chromium.
          Cloud-deployable with zero browser dependencies.
  Sacrificed: Dependency on external service, 3-8s latency per call,
              less control over what content Jina captures
  Verdict: Correct call — Playwright was removed entirely after confirming
           Jina covers all the JS-rendering use cases transparently

TRADE-OFF 5: Universal normalizer vs per-site flattening
  Chose: Universal flatten_record() applied at extraction + save time
  Gained: Any future site's nested data is handled automatically
  Sacrificed: Flattening heuristics may occasionally produce imperfect output
  Verdict: Right call — per-site band-aids would have been unmaintainable

TRADE-OFF 6: Removed 10-minute dedup check
  Was: Reject jobs with identical prompt+URL submitted within 10 minutes
  Removed: Users found it confusing and blocking legitimate re-scrapes
  Verdict: Dedup check is a UX policy decision, not a correctness requirement.
           The prompt_hash column is still stored if dedup is re-enabled later.

TRADE-OFF 7: SSE vs WebSockets
  Chose: SSE
  Gained: Simple HTTP, native browser EventSource, automatic reconnect,
          no upgrade handshake
  Sacrificed: Bidirectionality (not needed)
  SSE limit: 6 concurrent connections per domain in HTTP/1.1 browsers.
             HTTP/2 removes this limit (relevant for multi-tab usage).

TRADE-OFF 8: BackgroundTasks vs Celery
  Chose: BackgroundTasks
  Gained: Zero infrastructure, same process, trivial setup
  Sacrificed: Task persistence on restart, task visibility, multi-worker routing
  Mitigation: All job state in SQLite — restart marks in-flight jobs failed + retry
