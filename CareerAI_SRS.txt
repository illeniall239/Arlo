1. INTRODUCTION
--------------------------------------------------------------------------------
1.1 Purpose
    The purpose of this document is to define the functional and non-functional
    requirements for "CareerPilot AI", a multi-agent system designed to 
    automate and optimize the job application process. This system aggregates 
    job listings, matches them against user resumes using RAG, generates 
    tailored application materials, and tracks application status.

1.2 Scope
    CareerPilot AI will provide:
    - Hybrid job data ingestion (APIs + Ethical Scraping).
    - AI-driven resume-to-job matching (Semantic Similarity).
    - Automated cover letter generation tailored to specific job descriptions.
    - Application tracking dashboard (Kanban style).
    - Secure storage of user PII (Personally Identifiable Information).

    Out of Scope:
    - Direct submission of applications to external portals (due to CAPTCHA/ToS).
    - Guaranteed interview selection.
    - Real-time email inbox monitoring (initially manual upload of confirmations).

1.3 Definitions & Acronyms
    - RAG: Retrieval-Augmented Generation
    - PII: Personally Identifiable Information
    - JD: Job Description
    - ToS: Terms of Service
    - Celery: Distributed task queue system
    - Chroma: Vector database for embeddings

2. OVERALL DESCRIPTION
--------------------------------------------------------------------------------
2.1 User Classes
    - Job Seeker (Primary): Uploads resume, sets preferences, views dashboard.
    - System Admin (Secondary): Monitors system health, manages scraper limits.

2.2 Operating Environment
    - Backend: Python 3.10+, FastAPI
    - Frontend: Streamlit (Web Dashboard)
    - Database: PostgreSQL (Relational), ChromaDB (Vector)
    - Cache/Queue: Redis
    - Deployment: Docker Containers (Local/Cloud)
    - AI Models: Llama 3 (via Ollama) or OpenAI GPT-4o

2.3 Design Constraints
    - Scraping must respect robots.txt and rate limits (max 1 req/5s per domain).
    - PII must be encrypted at rest in PostgreSQL.
    - System must function in "Mock Mode" for demonstration without live APIs.
    - Latency for job matching must be < 5 seconds per job.

2.4 Assumptions
    - Users have a digital resume (PDF/Text).
    - External Job APIs (e.g., JSearch) remain accessible.
    - LLM API availability is stable.

3. SYSTEM FEATURES (FUNCTIONAL REQUIREMENTS)
--------------------------------------------------------------------------------
3.1 Module 1: Data Ingestion (Scraper & API)
    FR-1.1: System shall fetch job listings via configured APIs (e.g., JSearch).
    FR-1.2: System shall scrape company career pages if API data is insufficient.
    FR-1.3: System shall cache all fetched data in Redis for 24 hours to prevent 
            redundant requests.
    FR-1.4: System shall switch to "Mock Data Mode" if external services fail.
    FR-1.5: System shall normalize all job data into a standard JSON schema 
            (Title, Company, Description, Location, URL).

3.2 Module 2: AI Matching Engine (RAG)
    FR-2.1: System shall embed User Resume into Vector DB (Chroma) upon upload.
    FR-2.2: System shall embed Job Descriptions into Vector DB upon ingestion.
    FR-2.3: System shall calculate cosine similarity between Resume and Job embeddings.
    FR-2.4: System shall generate a "Match Score" (0-100%) for each job.
    FR-2.5: System shall provide a textual explanation for the score 
            (e.g., "Missing skill: AWS").

3.3 Module 3: Agent Orchestration (CrewAI)
    FR-3.1: System shall employ a "Matcher Agent" to validate semantic fit.
    FR-3.2: System shall employ a "Writer Agent" to generate cover letters.
    FR-3.3: System shall employ a "Tracker Agent" to update application status.
    FR-3.4: Agents shall run asynchronously via Celery workers to prevent API blocking.
    FR-3.5: System shall log all agent thoughts and actions for auditability.

3.4 Module 4: User Dashboard (Streamlit)
    FR-4.1: Users shall view jobs in a Kanban board (Interested, Applied, Interview, Rejected).
    FR-4.2: Users shall be able to filter jobs by Match Score (>80%, >50%, etc.).
    FR-4.3: Users shall be able to download generated Cover Letters (PDF/Txt).
    FR-4.4: Users shall be able to manually update application status.

3.5 Module 5: Security & Privacy
    FR-5.1: System shall encrypt sensitive fields (Phone, Address, Email) in DB.
    FR-5.2: System shall not store raw LLM responses containing PII in logs.
    FR-5.3: System shall provide a "Delete All Data" function for users.

4. EXTERNAL INTERFACE REQUIREMENTS
--------------------------------------------------------------------------------
4.1 User Interfaces
    - Web Dashboard (Streamlit): Responsive, clean, dark/light mode.
    - CLI Interface (Optional): For triggering scrapers manually.

4.2 API Interfaces
    - POST /api/jobs/scrape: Trigger job ingestion.
    - POST /api/resume/upload: Upload and embed resume.
    - GET /api/jobs/matched: Retrieve sorted list of jobs.
    - POST /api/apply/generate: Trigger cover letter generation.

4.3 Hardware Interfaces
    - None (Cloud/Containerized).

4.4 Communications Interfaces
    - HTTPS for all external communications.
    - Redis Protocol for internal task queue.

5. NON-FUNCTIONAL REQUIREMENTS
--------------------------------------------------------------------------------
5.1 Performance
    - NFR-1: Job ingestion rate: Minimum 10 jobs per minute.
    - NFR-2: Match Score calculation: < 2 seconds per job.
    - NFR-3: Cover Letter Generation: < 30 seconds per document.
    - NFR-4: System uptime: 99% during demo periods.

5.2 Scalability
    - NFR-5: System shall support horizontal scaling of Celery workers.
    - NFR-6: Database shall support up to 10,000 job listings without degradation.

5.3 Reliability
    - NFR-7: System shall retry failed API calls 3 times with exponential backoff.
    - NFR-8: System shall gracefully degrade to Mock Mode if external APIs fail.

5.4 Security
    - NFR-9: All API endpoints shall require authentication (API Key or Session).
    - NFR-10: Passwords (if implemented) shall be hashed using bcrypt.

5.5 Ethics & Compliance
    - NFR-11: Scraper shall respect robots.txt directives.
    - NFR-12: Scraper shall identify itself with a valid User-Agent string.
    - NFR-13: System shall not bypass CAPTCHAs or authentication walls.

6. AI/ML SPECIFIC REQUIREMENTS
--------------------------------------------------------------------------------
6.1 Model Performance
    - AI-1: Match Score correlation with human judgment > 85%.
    - AI-2: Cover Letter hallucination rate < 5% (verified by eval script).
    - AI-3: System shall cite specific JD sections when generating cover letters.

6.2 Data Privacy (AI)
    - AI-4: Resume data shall not be sent to public LLMs without user consent.
    - AI-5: Local models (Ollama) shall be used for PII-heavy processing where possible.

6.3 Evaluation & Monitoring
    - AI-6: System shall log token usage per task for cost monitoring.
    - AI-7: System shall include a "Feedback Loop" where users can rate 
            generated cover letters to fine-tune prompts.

7. DATA MODEL (HIGH LEVEL)
--------------------------------------------------------------------------------
7.1 Users
    - id (PK), email, password_hash, created_at

7.2 Resumes
    - id (PK), user_id (FK), file_path, embedding_vector, skills_extracted

7.3 Jobs
    - id (PK), source_url, title, company, description, embedding_vector, 
      scraped_at, is_cached

7.4 Applications
    - id (PK), user_id (FK), job_id (FK), status (Enum), cover_letter_path, 
      match_score, created_at

7.5 Audit Logs
    - id (PK), agent_name, action, timestamp, metadata_json

8.2 Risk Management
    - Risk: IP Ban from scraping.
      Mitigation: Use APIs first, implement Redis caching, use Mock Mode for demos.
    - Risk: LLM Cost Overrun.
      Mitigation: Token limits per task, use smaller models for classification.
    - Risk: PII Leak.
      Mitigation: Encryption at rest, local model processing for sensitive data.
